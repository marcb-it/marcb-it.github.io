Active Directory Administration Lab

Overview

This lab built a complete Windows Server Active Directory environment in a virtual lab and used it to practice the day-to-day administration tasks expected in a help-desk, desktop support, or junior sysadmin role. The goal was to move from "I know what these things are" to "I can do them confidently on demand."

What I configured

Installed Windows Server, set a static IP, promoted it to a domain controller for the lab.local domain, and configured DNS. Stood up a Windows 11 client, pointed its DNS at the DC, and joined the client to the domain.

Built an OU hierarchy with top-level OUs (Employees, Groups, Workstations, Service Accounts, Disabled Users) and department sub-OUs under Employees (IT, HR, Sales). Provisioned test users in each department OU and created security groups using a consistent GG_ naming convention covering department membership, VPN access, file-share access, and local admin rights — for example GG_IT_Users, GG_VPN_Users, GG_FileShare_HR_RW.

Practiced help-desk tasks against the lab: password resets, account unlocks (simulating lockouts from the client), enable/disable, OU moves to model department transfers, and group membership additions/removals. Worked through several ticket-style scenarios end-to-end.

Created a basic GPO linked to a department OU, configured a User Configuration policy (e.g., restrict Control Panel access), ran gpupdate /force on the client, and verified application with gpresult /r to see exactly which policies applied and in what order.

Executed end-to-end onboarding for a new Sales hire — account creation, OU placement, group assignment to Sales/VPN/file-share groups, and documentation — and the matching offboarding workflow: account disable, password reset, group removal, move to the Disabled Users OU with a termination date annotation.

What I learned

The lab connected concepts I use in production AD work — group-based access, OU organization, GPO scoping — to the underlying mechanics. Working through the onboarding and offboarding scenarios reinforced why naming conventions, group-based access (vs. direct user permissions), and a Disabled Users OU all matter for security and compliance. The ticket-style scenarios at the end were where confidence actually came from.

Microsoft 365 Administration Lab

Overview

This lab used a Microsoft 365 test tenant to practice user lifecycle and collaboration administration across all five M365 admin centers, with an emphasis on identity, licensing, and security tasks that recur in real sysadmin and help-desk-engineer work.

What I configured

Created test accounts in the Microsoft 365 admin center (Alex Admin as an admin-style user, Taylor User and Jordan User as standard employees) and assigned licenses covering Exchange Online, Teams, and SharePoint/OneDrive. Validated that Exchange mailboxes provisioned automatically once licensing was applied.

Configured shared mailboxes and Microsoft 365 Groups for collaboration testing. Worked through the Exchange admin center to view mailbox attributes and basic mail flow; the Teams admin center to review messaging and meeting policies; and the SharePoint admin center to review site-level permissions and external sharing controls.

Enforced MFA on test accounts through the Microsoft Entra admin center and reviewed sign-in activity to validate enforcement. Documented a secure onboarding and offboarding workflow that combines license assignment, MFA, group membership, and access deprovisioning.

What I learned

M365 is administered across multiple specialized portals, and knowing which task lives where matters. License assignment triggers mailbox provisioning automatically — that's not always obvious starting out. MFA enforcement from Entra is the single highest-impact security control in M365, and the lab made clear how identity (Entra) is the foundation everything else — Exchange, Teams, SharePoint — sits on top of.

Linux Server Fundamentals and Troubleshooting Lab

Overview

This lab built core Linux administration fluency on Ubuntu Server LTS, with a focus on the command-line skills that appear in every sysadmin and infrastructure support job posting that lists "Linux" as a requirement. Folded in a side-by-side comparison against Windows Server administration.

What I configured

Installed Ubuntu Server LTS in a VM and worked through filesystem navigation (cd, ls, pwd, find), user and group management (useradd, usermod, groupadd, passwd), and file permissions (chmod in both numeric and symbolic forms, chown for ownership changes).

Practiced package management with apt (update, upgrade, install, remove), service control with systemctl (start, stop, enable, status), and log review using journalctl and the contents of /var/log/.

Walked through a service-down troubleshooting scenario: identified the failed service via systemctl status, reviewed recent log entries with journalctl -u, checked dependencies, and resolved the issue.

Documented a side-by-side comparison of common administration tasks (user creation, service restart, network configuration check, log review) on Windows Server vs. Linux to support faster troubleshooting when switching between the two.

What I learned

Linux administration becomes much faster once the small set of core verbs (ls, cd, cat, less, tail, grep, find, systemctl, journalctl) becomes second nature. The cross-platform comparison clarified that the underlying administration tasks are the same on Windows and Linux — what changes is the tooling. That perspective makes it easier to support mixed environments instead of treating Linux as a foreign system.

PowerShell Scripting Lab

Overview

This lab built two PowerShell projects to develop scripting fluency beyond syntax memorization, focused on the data collection and CSV reporting workflows that come up in real IT admin work.

What I built

Set up a working folder at C:\PowerShellLab\ and validated the PowerShell version with $PSVersionTable. Worked through PowerShell foundations: variables, strings, integers, arrays, hashtables, foreach loops, if/else conditional logic, and parameter input.

Week 7 — System Inventory Script: Wrote a script using Get-CimInstance to collect hostname, IP address, OS version, and serial number from the local machine, then exported the results to a structured CSV via Export-Csv.

Week 8 — CSV-Driven Device Reporting: Built a more realistic project. Created a mock Devices.csv as input, then wrote DeviceReport.ps1 to Import-Csv the file, loop through each record, attempt to gather information per device, log success or failure with try/catch error handling, and Export-Csv the consolidated results to DeviceReportOutput.csv. Added inline comments throughout and produced a README.txt explaining script purpose, expected inputs and outputs, and troubleshooting notes.

What I learned

PowerShell is most useful when the scripts produce structured output you can hand back to a manager or feed into another process — that's what makes CSV input and output central rather than incidental. Error handling with try/catch turns brittle scripts into ones that can run unattended across many devices and still produce a usable report when individual items fail. Writing the README forced me to think about who else would read the script, which sharpened the inline comments.

Virtual Machine Deployment and Networking Lab

Overview

This lab developed virtualization administration fluency — deploying VMs, managing snapshots, and configuring virtual networking — with an emphasis on the troubleshooting patterns specific to virtualized environments.

What I configured

Deployed multiple VMs in a desktop hypervisor (VirtualBox / VMware Workstation), configured CPU and memory allocation appropriate to each guest OS, and exercised snapshot and revert workflows to practice non-destructive testing and recovery.

Configured virtual networking in both NAT and bridged modes, then walked through a connectivity troubleshooting scenario: a VM that couldn't reach the network. Worked through the isolation steps — adapter mode, IP assignment, DNS resolution, gateway reachability — and documented the resolution.

Captured snapshot/restore procedures and a brief virtual networking reference for repeatable lab and support use.

What I learned

Snapshots are the single feature that makes lab work efficient — being able to revert to a known good state means you can experiment without setup penalty. The NAT vs. bridged distinction matters constantly when a VM can or can't reach the host network, and developing a quick mental checklist for that situation saves real time in support work.

Cloud Administration Fundamentals Lab

Azure, Entra ID, Intune, and AWS

Overview

This lab built foundational cloud administration skills across Microsoft Azure and AWS, focused on identity, compute, storage, access control, and monitoring — the building blocks that show up in any IT administration or infrastructure support role with cloud exposure. Used Free Tier and trial services throughout to keep the lab cost-controlled.

What I configured — Microsoft Azure

Created and managed test users and groups in Microsoft Entra ID, enforced MFA, and reviewed Conditional Access concepts to understand how identity protection extends beyond simple authentication. Deployed a resource group, an Azure virtual network with subnets, and a Free Tier Azure VM (B-series), then reviewed role-based access control assignments at the subscription and resource group level.

Reviewed the Microsoft Intune admin center to understand cloud-based endpoint management: device enrollment concepts, compliance policies, and how Intune ties into a hybrid AD/Entra environment.

What I configured — AWS

Configured IAM users and groups with least-privilege policy attachments, launched a Free Tier EC2 instance (Amazon Linux), and reviewed instance connectivity via security group inbound rules. Created an S3 bucket and reviewed the default block-public-access settings. Reviewed CloudWatch metrics for the EC2 instance at a conceptual level.

Comparative documentation

Throughout the lab, captured architecture notes comparing how Azure and AWS solve the same five problems with different terminology: identity (Entra ID vs. IAM), compute (Azure VM vs. EC2), storage (Azure Storage vs. S3), network security (NSG vs. security group), and monitoring (Azure Monitor / Log Analytics vs. CloudWatch).

What I learned

The biggest takeaway was that Azure and AWS solve the same problems with different terminology — once you map "security group" to "NSG" and "EC2" to "Azure VM," moving between them becomes much easier. Free Tier and trial services are essential for hands-on learning without runaway cost, and the cleanup step at the end of each session matters more than I expected. Cloud identity (Entra, IAM) is the foundation everything else depends on — MFA, RBAC, and Conditional Access aren't peripheral, they're the gate.
